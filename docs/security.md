# Security

## The Docker socket problem

Any process that can talk to `/var/run/docker.sock` can become root on the
host. The path is short: spawn a privileged container, bind-mount `/`, write
an SSH key or a cron job, done. This is well-known. Most Docker dashboards and
proxies ignore it and mount the socket directly into the tool's container
anyway.

dockerdynomesh needs the Docker API — it has to watch events and inspect
containers. The question is how much of the API to expose to the code that
reads container metadata.

## How it's handled here

**socket-proxy sidecar.** The host socket (`/var/run/docker.sock`) is mounted
read-only into a single dedicated container: `tecnativa/docker-socket-proxy`.
That sidecar exposes only four API families on a private TCP socket on
`dynomesh-net`:

- `GET /events` — container and network events
- `GET /containers/*` — container list and inspect
- `GET /networks/*` — network membership
- `GET /info` — daemon info

All other endpoints are denied by default. Two consumers connect to
`tcp://socket-proxy:2375`: the **discoverer** (via `DOCKER_HOST`) and
**Traefik's docker provider** (via `providers.docker.endpoint` in
`traefik.yml`). Neither container has a path to the host socket — Traefik
in particular does not bind `/var/run/docker.sock`, which is the more common
configuration in upstream Traefik examples and the configuration this project
deliberately avoids.

**Nonroot discoverer.** The discoverer container runs as distroless nonroot
(UID 65532). Even if a bug in the discoverer allows arbitrary code execution,
the process has no write access to the host filesystem, no write access to the
Docker API (denied by socket-proxy), and runs in a network namespace shared
only with the other dynomesh containers.

**What a compromised discoverer can do:** read container metadata (names,
labels, network IPs, exposed ports). That's it. It cannot exec into
containers, pull images, start privileged containers, or reach the host
filesystem.

**What it cannot do:** traefik and certgen still run as root inside their
containers. The Docker socket hardening specifically targeted the one service
that had to talk to Docker. Traefik runs as root because the official image
does; certgen runs as root because mkcert writes to a root-owned CA directory.
Neither mounts the Docker socket.

## The root CA trust model

mkcert generates a self-signed root CA. Once you install it in your trust
store (system keychain on macOS, `update-ca-certificates` on Ubuntu), your
browser trusts any cert signed by that CA without warnings.

This means: anyone who has the CA private key (`rootCA-key.pem` in the `ca`
Docker volume) can issue certs for any domain and intercept your traffic. The
key is root-owned with mode 0600, accessible only inside the certgen container.
It does not leave the volume.

Practical limits of this model:

- The CA is machine-local. Don't copy the `ca` volume between machines, and
  don't share `root-ca.pem` + `rootCA-key.pem` together.
- If an attacker gets root on your Docker host, they have the CA key. This is
  the same threat model as any local-machine attack surface.
- The CA is not registered with any CT log and has no CRL. It is for local
  development use only.

## Why mkcert instead of Let's Encrypt

Let's Encrypt requires a public domain and outbound HTTP-01 or DNS-01
challenge verification. `docker.localhost` and other `.localhost` names aren't
publicly routable, so Let's Encrypt won't issue certs for them. mkcert is
explicitly built for this case: it generates a CA you install locally and uses
it to sign certs for whatever names you give it, including RFC-reserved names
like `*.localhost`.

## Tailscale

When Tailscale is detected, two things change:

1. The wildcard cert gains additional SANs for the tailnet hostname shape
   (`*.<project>.<machine>.<tailnet>.ts.net`).
2. Traefik binds on the Tailscale interface IP in addition to `127.0.0.1`.

The cert SANs are generated by the same mkcert-based CA. Peers on your
tailnet who connect to `<service>.<project>.<machine>.ts.net` are trusting
the same locally-installed CA. They need to have the CA installed in their
trust store too. The welcome page at `http://setup.<machine>.<tailnet>/`
delivers `rootCA.pem` and install instructions for that purpose.

Tailscale itself handles authentication and encryption between peers at the
network layer. dockerdynomesh adds application-layer TLS on top of that.

## Known limits

- **Traefik runs as root.** The official `traefik:v3.x` image doesn't support
  a nonroot UID out of the box. Traefik is the most exposed service (it
  handles all inbound TLS). This is a known gap; the hardening effort was
  specifically about the Docker socket path.
- **welcome runs as root** inside an isolated container with no external port
  binding. Its only exposure is through Traefik on the tailnet.
- **certgen runs as root** for the same reason — mkcert writes to a
  root-owned CA directory.
- **The CA is permanent.** mkcert generates a CA that's valid for around 10
  years. There's no automated rotation. If you want to rotate it, delete the
  `ca` volume and re-run `./bootstrap.sh up` (this also invalidates any
  existing trust installs).
- **No audit log.** certgen logs reissue requests but doesn't record which
  process connected to its socket. The socket is world-readable within the
  `run` volume (mode 0666) because the nonroot discoverer needs to connect.
  Tightening this would require the discoverer and certgen to share a Unix
  group, which complicates the distroless image.
